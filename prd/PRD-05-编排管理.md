## 2. 术语定义

| 术语 | 定义 |
|------|------|
| **Orchestration（编排）** | 多智能体为共同目标进行协作的过程与机制，定义了多个Agent之间的协作模式、任务分配策略和信息流转规则，是实现复杂任务自动化处理的核心能力 |
| **Workflow（工作流）** | 定义操作步骤的顺序、分支条件与依赖关系的逻辑骨架，由节点（Node）和边（Edge）组成，支持顺序执行、条件分支、循环迭代和并行批处理等控制结构 |
| **ACP（Agent Communication Protocol）** | 规范多智能体之间信息交换的通信标准，定义了Agent间的消息格式、路由规则和事件订阅机制，确保异构Agent之间的互操作性 |
| **A2A（Agent-to-Agent）** | Agent间的直接通信协议，支持服务发现、远程调用和事件订阅，使Agent能够跨网络边界进行协作 |
| **Agent（智能体）** | 具备自主决策和任务执行能力的智能实体，能够理解自然语言指令、调用工具（Capability）并与其他Agent协作完成复杂任务 |
| **Capability（能力）** | Agent可执行的操作单元，包括MCP工具调用、API接口调用和内部处理逻辑，是Agent对外暴露的功能接口 |
| **协调模式（Coordination）** | 基于可视化工作流画布定义的多Agent协作模式，通过逻辑节点（分支/循环/批量）和代理节点（物流/商品/价格/订单/库存）的组合编排任务执行流程 |
| **聊天模式（Chat）** | 基于对话式交互的Agent协作模式，用户通过自然语言描述任务需求，系统自动解析并分配给合适的Agent执行 |
| **MCP（Model Context Protocol）** | 模型上下文协议，定义了Agent与外部工具/资源之间的标准化交互接口，使Agent能够调用MCP Server提供的工具能力 |

## 3. 需求背景

### 3.1 问题陈述

在实际业务场景中，复杂任务往往需要多个Agent协同完成。例如，一个电商订单处理流程可能涉及物流Agent、库存Agent、价格Agent、订单Agent等多个角色的协作。当前面临以下核心问题：

1. **协作流程不透明**：多Agent之间的任务分配和信息流转缺乏可视化的编排工具，协作逻辑难以理解和维护
2. **编排方式单一**：仅支持固定的工作流模式，无法适应需要灵活对话式交互的场景
3. **Agent配置与编排耦合**：Agent的创建和编排流程混杂在一起，导致配置复杂度高、复用性差
4. **缺乏预览和调试能力**：编排完成后无法在实际运行前预览效果，调试成本高
5. **跨系统协作困难**：缺乏标准化的Agent间通信协议（A2A/ACP），异构系统间的Agent协作难以实现

### 3.2 目标用户

| 用户角色 | 使用场景 |
|----------|----------|
| 业务架构师 | 设计多Agent协作流程，定义任务分配策略和信息流转规则 |
| Agent开发者 | 在编排中配置Agent参数，调试工作流逻辑 |
| 平台管理员 | 管理编排生命周期，监控编排运行状态 |
| 业务运营人员 | 通过聊天模式与编排交互，执行日常业务任务 |

### 3.3 业务目标

- 提供可视化工作流画布，支持拖拽式编排多Agent协作流程
- 支持协调模式（Coordination）和聊天模式（Chat）两种编排方式，覆盖结构化和非结构化场景
- 实现编排内Agent的独立配置，支持Agent的创建和参数调整
- 提供客户端预览和代码预览能力，支持编排效果的快速验证
- 支持A2A协议配置，实现跨系统Agent协作

**编排降级策略（模型 UNHEALTHY 级联）**：

当编排中包含的 Agent 因关联模型进入 UNHEALTHY 状态而降级或暂停时，编排须执行以下降级策略，确保与 PRD-04 §5.3 级联通知流程和 PRD-06 §6.1 Agent 降级处理策略对齐：

| 场景 | 受影响 Agent 位置 | 编排处理策略 | 编排结果状态 | 说明 |
|------|-------------------|-------------|-------------|------|
| 部分Agent降级（已配置备选模型） | 非关键路径 | 降级 Agent 继续使用备选模型执行，编排正常运行 | Running（降级运行） | 降级 Agent 的推理质量可能下降，编排结果中标注降级节点 |
| 部分Agent降级（已配置备选模型） | 关键路径 | 降级 Agent 继续使用备选模型执行，编排正常运行，但标记为降级 | Running（降级运行） | 关键路径降级需在编排执行结果中明确标注 |
| 部分Agent暂停（未配置备选模型） | 非关键路径 | 暂停 Agent 跳过执行，其余节点继续运行 | PartialSuccess | 非关键路径节点失败不影响整体流程，结果中标注跳过节点 |
| 部分Agent暂停（未配置备选模型） | 关键路径 | 编排暂停执行，等待 Agent 恢复或管理员干预 | Paused | 关键路径 Agent 不可用时编排无法继续，通知管理员 |
| 多个Agent依赖同一UNHEALTHY模型 | 含关键路径 | 按上述规则判断：若关键路径 Agent 有备选模型则降级运行，否则编排暂停 | PartialSuccess 或 Paused | 同一模型故障影响多个 Agent 时，统一处理避免重复通知 |
| 多个Agent依赖同一UNHEALTHY模型 | 均为非关键路径 | 受影响 Agent 均按各自降级策略处理，编排继续运行 | PartialSuccess | 合并通知，避免通知风暴 |

**编排降级决策流程**：
1. 编排接收到 Agent 降级/暂停通知后，评估受影响 Agent 在编排中的位置（关键路径 vs 非关键路径）；
2. 根据上表策略确定编排结果状态；
3. 若编排进入 Paused 状态，系统发送通知至管理员（站内消息 + 邮件），通知内容包含：编排 ID、编排名称、暂停原因、受影响 Agent 列表、建议操作；
4. 降级决策须在 Agent 降级/暂停通知到达后 **60 秒内** 完成。

**编排恢复策略**：
- 模型恢复 ACTIVE 后，受影响的 Agent 自动恢复（详见 PRD-06 §6.1 恢复策略），编排自动恢复执行，**无需重新调度**；
- 正在执行的任务从暂停点继续推进，新任务正常接收；
- 编排状态从 Paused 自动恢复为 Running，从 PartialSuccess 的降级运行自动恢复为正常 Running；
- 所有降级/恢复事件须记录审计日志，包含：编排 ID、Agent 降级详情、编排状态变更时间戳、恢复时间戳。

    end note

        - End节点输出最终结果
    end note

        - 关键节点执行失败

    end note

    note right of Timeout
        执行超时

        - [已确认] 默认300秒 —— 依据：复杂工作流需要较长执行时间
    end note

        部分成功
        - [已确认] 非关键路径节点失败 —— 依据：Batch节点中部分Agent失败但满足批处理策略
    end note

**工作流执行状态规则**：

| 规则编号 | 规则描述 |
|----------|----------|
| ORC-EX-001 | 工作流执行全局超时阈值默认为300秒，可通过编排Constraint配置覆盖 |
| ORC-EX-002 | 单个Agent节点执行超时由Call Timeout配置决定，默认30秒 |
| ORC-EX-003 | 失败重试次数由Retry Count配置决定，默认3次，重试间隔采用指数退避策略（1s/2s/4s） |
| ORC-EX-004 | [已确认] Pending状态最大等待时间为60秒，超时后自动标记为Failed —— 依据：避免资源长时间占用 |
| BR-05-021 | PartialSuccess 状态——当工作流中非关键节点失败但关键节点全部成功时，工作流标记为 PartialSuccess，结果聚合策略为"关键节点结果保留+非关键节点错误记录"。产品决策：PartialSuccess 视为成功（可查看部分结果），但需在结果中标注失败的节点列表。 |

**PartialSuccess 消费方处理定义**：

| 消费方 | 处理策略 | 说明 |
|--------|----------|------|
| 前端展示 | 工作流状态标签显示为"部分成功"（黄色/橙色标识），与"成功"（绿色）和"失败"（红色）形成视觉差异；结果面板分区展示：成功节点结果正常展示 + 失败节点折叠展示（含错误原因与重试入口） | 用户可直观区分完整成功与部分成功，并快速定位失败节点 |
| 下游流程触发 | PartialSuccess 视为成功，允许触发下游流程；下游流程仅接收成功节点的输出作为输入，失败节点的输出不参与下游数据传递 | 保证下游流程数据完整性，避免因部分节点失败导致级联错误 |
| 事件通知 | 发送 `workflow.executed` 事件（payload 中 `status=PartialSuccess`，`failed_nodes` 字段列出失败节点 ID 与错误信息） | 订阅方可根据 `status` 差异化处理 |
| 监控告警 | PartialSuccess 计入工作流成功率统计（分子+1），同时单独统计 PartialSuccess 比率，比率超过阈值时触发 P2 告警 | 兼顾成功率指标与异常趋势感知 |

> **超时层级说明**：工作流超时分为三个层级，由内到外逐级覆盖：(1) **节点超时**（默认60s，BR-05-012）——单个节点的最大执行时间，超时后节点标记为 TIMEOUT；(2) **全局超时**（默认300s，ORC-EX-001）——从工作流启动起的整体超时，超时后所有运行中节点取消；(3) **总执行超时**（DDL max_duration_s 默认600s）——包含重试在内的总执行时间上限。三层关系：节点超时 ≤ 全局超时 ≤ 总执行超时，违反此约束的配置在创建时被拒绝。

**后置条件**：
- 列表页面正确展示所有编排的核心字段信息
- 支持按字段排序和分页浏览

- **异常B - 权限不足**：页面展示"无权限访问"提示

- 删除操作需二次确认弹窗
- Copy操作成功后展示"已复制"提示，2秒后自动消失

- AC2：Status字段使用颜色编码区分三种状态（Published=#4CAF50绿色、Archived=#9E9E9E灰色、Draft=#FFC107黄色）

### 7.4 创建编排 - 步骤2.1：协调模式

**用户故事**：作为业务架构师，我希望通过可视化工作流画布设计多Agent协作流程，以便直观地定义任务分配逻辑和信息流转规则。

> **INVEST分析**：Independent（独立步骤）、Negotiable（节点类型可扩展）、Valuable（核心编排能力）、Estimable（工作量较大）、Small（拆分为画布操作和节点配置）、Testable（工作流输出可验证）

**前置条件**：
- 用户已完成步骤1基本信息配置
- 用户在步骤2选择了"协调模式"

**后置条件**：
- 工作流定义通过合法性校验
- 工作流节点和连接关系持久化存储

**主流程**：
1. 系统展示可视化工作流画布，画布左侧展示节点面板
2. 节点面板分为两个区域：

   **A. 逻辑节点区域**
   - **Branch（分支节点）**：条件分支，根据条件判断走不同路径
     - 配置项：条件表达式（支持变量引用）、分支数量（2-N）
   - **Loop（循环节点）**：循环执行，重复执行子流程直到满足退出条件
     - 配置项：循环条件表达式、最大循环次数（默认10次，上限100次）
   - **Batch（批量节点）**：并行批处理，将任务分发给多个Agent并行执行
     - 配置项：批处理策略（全部成功/任一成功/多数成功）、超时策略

   **B. 代理节点区域**
   以下代理节点为 E-commerce 类型编排的预设模板示例。实际代理节点应根据编排 Type 动态展示对应场景的预设模板，用户也可从 Agent 列表中自由选择。
   - **Shipping（物流Agent）**：处理物流相关任务
   - **Product（商品Agent）**：处理商品信息相关任务
   - **Price（价格Agent）**：处理价格计算和查询任务
   - **Order（订单Agent）**：处理订单管理任务
   - **Inventory（库存Agent）**：处理库存查询和更新任务
   - **Custom Agent（自定义Agent）**：用户自定义的Agent节点

3. 用户从节点面板拖拽节点到画布上
4. 用户通过连线工具连接节点，定义执行顺序
5. 用户点击节点打开配置面板，设置节点参数
6. 用户可调整节点位置、连线关系
7. 画布支持缩放（放大/缩小/适应画布）和平移
8. 用户点击"下一步"进入步骤3

**分支流程**：
- **分支A - 新建MCP Server**：画布工具栏提供"新建MCP Server"快捷按钮，点击后弹出MCP Server配置弹窗（复用能力管理模块的配置表单），配置完成后工具自动加载到当前编排
- **分支B - 节点配置代理**：点击代理节点后，如该Agent尚未创建，系统引导用户进入步骤3创建Agent
- **分支C - 工作流校验失败**：点击"下一步"时系统校验工作流完整性（如是否有未连接的节点、是否有孤立节点），校验失败时高亮问题节点并提示

**异常流程**：
- **异常A - 画布操作异常**：节点拖拽失败或连线异常，系统自动恢复到上一个稳定状态
- **异常B - 工作流保存失败**：系统提示"保存失败"，保留画布当前状态

**交互说明**：
- 画布采用无限画布设计，支持鼠标滚轮缩放和拖拽平移
- 节点使用不同颜色区分类型（逻辑节点-蓝色，代理节点-绿色）
- 连线使用带箭头的有向边，支持直线和折线两种样式
- 节点面板支持搜索过滤
- 画布工具栏提供：撤销/重做、对齐工具、自动布局、清空画布
- 右键菜单支持：删除节点、复制节点、查看节点配置

**验收标准**：

### 8.1 Orchestration与Agent的完整协作关系

```mermaid
flowchart LR
    subgraph Orchestration
        direction TB
        O1[编排基本信息]
        O2[协作模式定义]
        O3[全局变量]
    end

    subgraph Workflow
        direction TB
        W1[逻辑节点]
        W2[代理节点]
        W3[数据边]
    end

    subgraph Agent1[Agent 实例]
        direction TB
        A1[基本属性]
        A2[能力配置 Capability]
        A3[内部工作流]
        A4[A2A配置 Remote]
    end

    subgraph Agent2[Agent 实例N]
        direction TB
        AN1[基本属性]
        AN2[能力配置 Capability]
        AN3[内部工作流]
        AN4[A2A配置 Remote]
    end

    O2 --> W2
    W2 --> A1
    W2 --> AN1
    W3 --> A3
    W3 --> AN3
    A2 --> |调用MCP工具| MCP[MCP Server]
    AN2 --> |调用MCP工具| MCP
    A4 --> |A2A协议| Remote[远程Agent]
    AN4 --> |A2A协议| Remote
```

**协作关系说明**：

| 关系维度 | 说明 |
|----------|------|
| 编排→工作流 | 一个Orchestration包含一个Workflow定义（协调模式）或一个Chat配置（聊天模式） |
| 编排→Agent | 一个Orchestration可包含1-20个Agent实例 |
| 工作流→Agent | Workflow中的代理节点引用编排内的Agent实例，通过代理节点调用Agent |
| Agent→Capability | Agent通过Capability Type标签声明其能力，内部工作流中的工具调用节点引用具体的MCP工具 |
| Agent→Agent | Agent间通过Workflow的数据边传递信息，Remote类型Agent通过A2A协议跨系统通信 |
| [已确认] 编排→全局变量 | 编排级别的全局变量在所有Agent间共享，Agent内部变量仅在Agent内部工作流中有效 —— 依据：工作流节点间需要跨Agent传递数据 |

### 8.2 工作流节点间的数据传递机制（变量映射）

**变量体系**：

| 变量类型 | 作用域 | 定义位置 | 引用格式 | 说明 |
|----------|--------|----------|----------|------|
| 编排全局变量 | 整个编排内所有Agent | 编排Constraint配置 | `${orchestration.variable_name}` | [已确认] 跨Agent共享的配置参数，如API密钥、环境标识 —— 依据：多Agent协作需要共享上下文信息 |
| 工作流变量 | 单个编排工作流内 | 步骤2.1工作流画布 | `${workflow.variable_name}` | 编排工作流节点间传递的中间数据 |
| Agent内部变量 | 单个Agent内部工作流内 | 步骤4工作流画布 | `${agent.variable_name}` | Agent内部节点间传递的中间数据 |
| 节点输出变量 | 当前节点下游 | 节点配置面板 | `${node.node_id.output_key}` | 上游节点的输出作为下游节点的输入 |
| 系统上下文变量 | 整个编排内 | 系统预定义 | `${system.request_id}` / `${system.timestamp}` | [已确认] 系统自动注入的运行时上下文 —— 依据：工作流执行需要追踪和关联请求 |

**节点间数据映射规则**：

1. **输入映射**：每个节点可配置输入参数的来源，支持以下映射方式：
   - 直接引用上游节点输出：`${node.upstream_id.result}`
   - 引用工作流变量：`${workflow.variable_name}`
   - 引用编排全局变量：`${orchestration.variable_name}`
   - 静态值：直接输入常量值

2. **输出映射**：每个节点执行完成后，其输出自动注册为节点输出变量，下游节点可引用

3. **类型转换**：[已确认] 当变量类型不匹配时，系统自动进行隐式类型转换（如Number→String），转换失败时标记为节点执行错误 —— 依据：不同Agent的输入输出Schema可能不一致

4. **变量冲突**：[已确认] 同名变量按作用域优先级解析：节点输出 > 工作流变量 > 编排全局变量 > 系统上下文变量 —— 依据：最小作用域优先原则

### 8.3 Convert to Workflow的AI分析逻辑

**AI分析流程**：

| 阶段 | 输入 | 处理逻辑 | 输出 |
|------|------|----------|------|
| 1. 意图识别 | 任务描述文本 | NLP解析任务类型、目标和约束条件 | 任务意图标签、关键实体 |
| 2. Agent拆解 | 任务意图 + 关键实体 | 根据任务描述识别所需Agent角色，匹配系统已有Agent或建议新建 | Agent列表及角色定义 |
| 3. 流程编排 | Agent列表 + 任务意图 | [已确认] 基于任务语义推断Agent间的执行顺序和依赖关系，使用拓扑排序生成DAG —— 依据：工作流本质上是有向无环图 | 工作流节点序列和连接关系 |
| 4. 变量映射 | Agent能力描述 + 工作流结构 | [已确认] 分析Agent的输入输出Schema，推断节点间的数据传递关系，自动生成变量映射 —— 依据：Agent间需要传递结构化数据 | 变量映射配置 |
| 5. 条件分支识别 | 任务描述中的条件语义 | 识别"如果...则...否则..."等条件语句，生成Branch节点 | 条件分支节点及条件表达式 |

**AI分析约束**：

| 约束项 | 说明 |
|--------|------|
| 最大分析时间 | 30秒，超时后提示用户重试 |
| 最小任务描述长度 | 10字符，低于此长度提示补充 |
| [已确认] 生成工作流节点数上限 | 20个节点 —— 依据：避免生成过于复杂的工作流，超出用户理解范围 |
| 用户确认 | 转换结果必须经用户确认后方可保存，用户可手动调整 |
| 原始数据保留 | 转换后保留原始聊天配置，支持回退到聊天模式 |

**A2A通信规则**：

| 规则编号 | 规则描述 |
|----------|----------|
| A2A-COM-001 | 握手请求必须携带AppID和签名（AppKey + Timestamp的HMAC-SHA256签名） |

| A2A-COM-003 | 任务调用支持同步和异步两种模式，默认使用异步模式 |

### 8.5 编排执行的超时和重试策略

**超时层级**：

| 超时层级 | 配置位置 | 默认值 | 说明 |
|----------|----------|--------|------|
| 全局执行超时 | 编排Constraint | 300秒 | 整个工作流执行的最大时长，超时后强制终止 |
| Agent调用超时 | Agent Call Timeout | 30秒 | 单个Agent节点执行的最大时长 |
| A2A握手超时 | Agent Handshake Timeout | 10秒 | A2A远程Agent握手的最大等待时长 |
| [已确认] 节点间传递超时 | 工作流配置 | 60秒 | 节点间数据传递的最大等待时长 —— 依据：大数据量传递可能需要较长时间 |
| [已确认] Pending等待超时 | 系统配置 | 60秒 | 工作流在Pending状态的最大等待时长 —— 依据：避免资源长时间占用 |

**重试策略**：

| 重试场景 | 触发条件 | 重试次数 | 重试间隔 | 说明 |
|----------|----------|----------|----------|------|
| Agent调用重试 | Agent执行失败（非业务逻辑错误） | Retry Count配置（默认3次） | 指数退避（1s/2s/4s） | 仅对可重试错误（网络超时、服务不可用）进行重试 |
| A2A连接重试 | A2A握手失败 | 3次 | 指数退避（1s/2s/4s） | [已确认] 连接级别的重试独立于调用重试 —— 依据：连接失败和调用失败是不同层级的问题 |
| [已确认] 工作流节点重试 | 非关键路径节点失败 | 1次 | 固定2秒 | 仅在Batch节点的非关键路径上重试 —— 依据：批处理场景允许部分失败 |
| 全局重试限制 | 工作流整体执行失败 | 不自动重试 | 无 | [已确认] 全局失败需人工介入或通过外部调度器触发 —— 依据：全局失败可能涉及业务逻辑错误，不宜自动重试 |

**超时处理流程**：

1. 全局执行超时 → 强制终止所有运行中节点 → 工作流状态标记为Timeout → 记录超时日志
2. Agent调用超时 → 当前节点标记为Failed → 根据Retry Count决定是否重试 → 重试耗尽后工作流状态取决于节点是否在关键路径
3. A2A握手超时 → 当前Agent标记为不可用 → [已确认] 降级为本地模式或使用缓存的历史能力列表 —— 依据：风险预案ORC-RISK-003

### 8.8 协调模式与聊天模式的适用场景对比

| 对比维度 | 协调模式（Coordination） | 聊天模式（Chat） |
|----------|--------------------------|-------------------|
| **定义方式** | 可视化工作流画布，拖拽节点和连线 | 自然语言任务描述，对话式交互 |
| **适用场景** | 流程固定、逻辑清晰的结构化任务 | 流程灵活、需求模糊的探索性任务 |
| **典型用例** | 电商订单处理、库存同步、价格计算 | 客户咨询、智能客服、需求探索 |
| **编排精度** | 高，可精确控制每个节点的执行逻辑 | 中，依赖AI解析任务描述，可能存在歧义 |
| **学习成本** | 较高，需理解工作流概念和节点类型 | 较低，自然语言描述即可 |
| **可调试性** | 强，可逐步执行、查看节点状态 | 弱，对话过程较难追踪内部逻辑 |
| **可复用性** | 高，工作流模板可直接复用 | 中，需重新描述或转换为工作流后复用 |
| **灵活性** | 低，修改需重新编辑工作流 | 高，可通过对话动态调整 |
| **可预测性** | 高，执行路径明确 | 低，AI解析结果可能因描述差异而不同 |
| **模式转换** | 不可转换为聊天模式 | 可通过Convert to Workflow转换为协调模式 |
| **目标用户** | 业务架构师、Agent开发者 | 业务运营人员 |
| **配置复杂度** | 高（需配置节点、连线、变量映射） | 低（仅需任务描述） |
| **执行效率** | 高，预定义路径减少运行时决策 | [已确认] 较低，运行时需AI解析和路由 —— 依据：聊天模式每次执行都需要AI理解意图并分配Agent |

**模式选择建议**：

- **优先选择协调模式**：当任务流程明确、步骤固定、需要精确控制执行逻辑时
- **优先选择聊天模式**：当任务需求模糊、流程灵活、需要快速验证协作方案时
- **推荐工作流**：聊天模式验证 → Convert to Workflow转换 → 协调模式精细化 → 部署上线

### 16.2 与其他模块的关系

**关系1：协同扩展层内部 — Orchestration ↔ ACP**

| 维度 | 说明 |
|------|------|
| 方向 | Orchestration ↔ ACP |
| 类型 | 协同依赖 |
| 描述 | 编排管理基于 ACP（Agent Communication Protocol）实现 Agent 间的消息路由、事件订阅和远程调用。编排内的 Remote 类型 Agent 通过 ACP 与外部系统 Agent 通信 |
| 数据流 | 编排执行 → ACP 消息分发 → 远程 Agent → 结果回传 |
| 影响范围 | ACP 通信故障时，编排内的 Remote 类型 Agent 无法执行，影响跨系统协作场景 |

**关系2：协同扩展层 → 编排翻译层 — Orchestration → Workflow**

| 维度 | 说明 |
|------|------|
| 方向 | Orchestration → Workflow |
| 类型 | 调度依赖 |
| 描述 | 编排管理通过 Workflow 定义任务执行步骤，调用工作流引擎执行。编排工作流画布（步骤 2.1）和 Agent 内部工作流画布（步骤 4）均依赖 Workflow 模块 |
| 数据流 | 编排定义 → Workflow 实例化 → 节点执行 → 结果聚合 |
| 影响范围 | Workflow 模块故障时，编排的执行能力降级 |

**关系3：协同扩展层 → 核心驱动层 — Orchestration → Agent**

| 维度 | 说明 |
|------|------|
| 方向 | Orchestration → Agent |
| 类型 | 调度依赖 |
| 描述 | 编排管理调度 Agent 执行任务，编排内可包含 1-20 个 Agent 实例。Agent 通过 LLM 进行推理决策，编排负责任务分配和结果聚合 |
| 数据流 | 编排调度 → Agent 选择 → LLM 推理 → 结果返回 → 流程推进 |
| 影响范围 | Agent 下线或配置变更时需评估对编排流程的影响 |

**关系4：协同扩展层 → 能力构建层 — Orchestration → Capability**

| 维度 | 说明 |
|------|------|
| 方向 | Orchestration → Capability |
| 类型 | 资源依赖 |
| 描述 | 编排内的 Agent 通过 Capability 调用 MCP 工具、API 接口等能力资源。编排工作流画布提供"新建 MCP Server"快捷入口，配置完成后工具自动加载到当前编排 |
| 数据流 | Agent 节点 → Capability 调用 → MCP Server → 结果返回 |
| 影响范围 | Capability 不可用时，编排内依赖该能力的 Agent 节点执行失败 |

| 规范项 | 说明 | 参考 |
|--------|------|------|
| 协议 | GraphQL over HTTP(S),WebSocket(用于 Subscription/AG-UI 实时推送) | PRD-00 §4 |

| 限流 | 单租户 100 QPS，超出由网关层返回 HTTP 200 + 业务错误码 055429（限流触发）(GraphQL `errors[]` 携带 `THROTTLED`) |

| 编号 | 验收标准 | 验证方法 |
|------|----------|----------|
| AC-API-ORC-01 | 编排管理所有 API 遵循 GraphQL 统一规范(POST `/graphql`) | GraphQL Schema 审查 |
| AC-API-ORC-02 | 所有操作通过 §A5 列出的 Query / Mutation / Subscription 入口 | Schema 审查 |
| AC-API-ORC-03 | 所有 API 响应格式符合统一规范（成功/分页/错误） | 接口自动化测试 |
| AC-API-ORC-04 | 错误码体系正确使用(`BIZ_ORCHESTRATION_*` 命名空间 + 6 位 `055001-055999` 段位,业务层统一 HTTP 200) | 接口测试 |
| AC-API-ORC-05 | 分页接口遵循 Relay Connection 规范(first/after/last/before/cursor/pageInfo/totalCount) | 接口测试 |
| AC-API-ORC-06 | 所有 Mutation 接受 `idempotencyKey: ID!` 实现幂等 | 接口测试 |
| AC-API-ORC-07 | 接口限流 100 QPS/租户，超出由网关层返回 HTTP 200 + 业务错误码 055429（限流触发） | 压力测试 |

## 19. 跨模块事务一致性

本章定义编排管理模块在跨模块联动时的事务一致性约束，避免因 Agent / Knowledge / Tool / LLM / Workflow 等子模块的删除、禁用或状态变更导致编排处于"悬挂引用"状态。

### 19.1 事务一致性原则

| 原则编号 | 原则名称 | 原则描述 |
|----------|----------|----------|
| TX-ORC-001 | 强引用外键 | 编排(Orchestration)与 Agent、Workflow、Knowledge、MCP Tool、LLM Model 之间建立强引用外键(`ON DELETE RESTRICT`),被引用的资源不可被物理删除,仅可软删除 |
| TX-ORC-002 | 软删除传播 | 当被引用的资源(Agent、Knowledge、MCP Tool)被软删除时,编排自动将该引用标记为"已失效",并在保存时拒绝发布 |
| TX-ORC-003 | 状态联动校验 | 编排发布(Published)前必须校验所有引用的 Agent / Knowledge / Tool / LLM 状态均为 Active,否则返回 055101~055106 业务错误码(业务层 HTTP 200) |
| TX-ORC-004 | 跨模块事务模式 | 编排保存涉及多个子模块时采用 Saga 模式(含补偿事务)或 Outbox 模式,任一子事务失败触发补偿回滚 |
| TX-ORC-005 | 租户一致性 | 编排与被引用的所有资源必须属于同一 partition_key,跨租户引用返回 055201(业务层 HTTP 200) |

### 19.2 悬挂引用检测流程

```mermaid

    A[编排发布前/执行前触发检测] --> B[扫描编排引用的所有外部资源]

    B --> B2[引用的 Knowledge]
    B --> B3[引用的 MCP Tool]
    B --> B4[引用的 LLM Model]

    B2 --> C
    B3 --> C
    B4 --> C
    B5 --> C

    C -->|软删除| D1[标记为已失效<br/>错误码 5510x]
    C -->|Archived| D2[标记为已失效<br/>错误码 055102/055105]
    C -->|Published| E[引用有效]

    D1 --> F[编排发布被阻止<br/>返回错误码列表]
    D2 --> F

    style D1 fill:#FFB6C1

```

### 19.3 跨模块错误码与场景对照表

> 跨模块错误码使用 `BIZ_ORCHESTRATION_*` 命名空间(数字段位 6 位 `055001-055999`),详见 §18.2。

| 错误码 | 场景 | 编排侧处理 |
|--------|------|-----------|
| 055101 | 引用已删除的 Agent | 编排自动从节点中移除该 Agent 引用,提示用户重新配置 |
| 055102 | 引用已禁用的 Agent | 编排节点标记为失效,发布时阻止 |
| 055103 | 引用已删除的 Knowledge | 编排 Agent 节点标记为失效,发布时阻止 |
| 055104 | 引用已删除的 MCP Tool | 编排 Agent 节点标记为失效,发布时阻止 |
| 055105 | 引用已禁用的 LLM | 编排 Agent 节点标记为失效,发布时阻止 |
| 055106 | 引用已删除的 Workflow | 编排标记为失效,不可执行 |

### 19.4 验收标准

| 编号 | 验收标准 | 验证方法 |
|------|----------|----------|
| AC-TX-ORC-01 | 编排引用的 Agent 软删除后,编排发布时返回 055101 业务错误码(业务层 HTTP 200) | 软删除被引用 Agent,尝试发布编排 |

## 22. ACP（Agent Communication Protocol）安全规范

> **协议边界**：ACP（Agent Communication Protocol）用于跨域 Agent 通信，与 A2A（智能体对智能体，见 [PRD-06 §7.6](../PRD-06-智能体管理.md)）、AG-UI（前端事件契约，见 [PRD-00 §10](../PRD-00-平台总览与全局规范.md)）严格区分；三者的适用场景、消息格式、安全模型与协议边界详见 [PRD-00 §10.6](../PRD-00-平台总览与全局规范.md)。
>
> | 协议 | 范围 | 方向 | 主要用途 |
> |------|------|------|----------|
> | ACP | Agent ↔ Agent（跨域） | 双向 | 跨域 Agent 任务派发与回执 |
> | A2A | Agent ↔ Agent（同域） | 双向 | 同域多 Agent 协同 |
> | AG-UI | 前端 ↔ 后端 | 服务端推送 | 编排执行状态实时可视化 |

本章定义 Agent 间通信协议（ACP）的安全规范，包括身份认证、防重放、消息完整性、机密性保护。

### 22.1 ACP 协议栈

```mermaid
flowchart TB
    subgraph 应用层
        A1[业务消息: 任务请求/响应/事件]
    end

    subgraph 协议层
        B1[消息头: AppID/Nonce/Timestamp/Version]
        B2[消息体: 业务 JSON]
        B3[签名: HMAC-SHA256]
    end

    subgraph 传输层
        C1[HTTPS/TLS 1.2+]
    end

    A1 --> B2
    B1 --> A1
    B3 --> B2
    B2 --> C1
    B1 --> C1
```

### 22.2 身份认证流程

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 客户端生成请求 | 携带 `AppID` + `Timestamp`（ISO 8601） + `Nonce`（UUID） + `Body`（业务 JSON） |
| 2 | 客户端计算签名 | `Signature = HMAC-SHA256(AppKey, AppID + "\n" + Timestamp + "\n" + Nonce + "\n" + Body)` |
| 3 | 客户端发起请求 | HTTP Header: `X-App-Id`、`X-Timestamp`、`X-Nonce`、`X-Signature` |
| 4 | 服务端验证时间戳 | 时钟偏移 ≤ 300 秒,否则拒绝(业务错误码 055603) |
| 5 | 服务端验证 Nonce | 查询 Nonce 集合(Redis,TTL=10分钟),重复则拒绝(业务错误码 055604) |
| 6 | 服务端重新计算签名 | 用本地存储的 AppKey 计算签名并比对,不匹配则拒绝(业务错误码 055601) |
| 7 | 服务端颁发 Session Token | 通过认证后返回 Session Token，有效期 24 小时 |
| 8 | 后续请求 | 携带 `Authorization: Bearer {SessionToken}` |

### 22.3 安全规则

| 规则编号 | 规则名称 | 规则描述 |
|----------|----------|----------|
| ACP-SEC-001 | 强制 HMAC 签名 | 所有 ACP 消息必须携带 HMAC-SHA256 签名 |
| ACP-SEC-002 | 时钟偏移限制 | 客户端与服务端时钟偏移不得超过 300 秒 |
| ACP-SEC-003 | Nonce 防重放 | Nonce 必须全局唯一，TTL=10 分钟，重复 Nonce 直接拒绝 |
| ACP-SEC-004 | Session Token 有效期 | Session Token 有效期 24 小时，过期后必须重新握手 |
| ACP-SEC-005 | AppKey 加密存储 | AppKey 在数据库中 AES-256 加密存储，展示时脱敏 |
| ACP-SEC-006 | TLS 加密传输 | 所有 ACP 通信强制 HTTPS/TLS 1.2+ |
| ACP-SEC-007 | 错误码规范 | 认证失败相关业务错误码使用 055601~055605 段位(业务层 HTTP 200) |

### 22.4 验收标准

| 编号 | 验收标准 | 验证方法 |
|------|----------|----------|

## 23. MCP Server 健康检查

本章定义 MCP Server 的健康检查规范，确保编排执行时能够及时检测 MCP Server 故障并自动降级。

### 23.1 健康检查规范

| 配置项 | 默认值 | 可配置范围 | 说明 |
|--------|--------|------------|------|
| 检查间隔 | **30 秒** | 10~300 秒 | 健康检查的固定周期 |
| 检查超时 | 5 秒 | 1~30 秒 | 单次健康检查的超时时间 |
| 连续失败阈值 | 3 次 | 1~10 次 | 连续失败达到阈值后标记为不可用 |
| 恢复探测间隔 | 60 秒 | 30~600 秒 | 标记不可用后的恢复探测间隔 |
| 检查端点 | `/health` | 任意 URL | MCP Server 提供的健康检查端点 |
| 检查方法 | GET | GET/POST | 健康检查 HTTP 方法 |

### 23.2 健康状态机

```mermaid
stateDiagram-v2
    [*] --> Unknown : MCP Server 注册
    Unknown --> Healthy : 首次健康检查成功
    Healthy --> Healthy : 健康检查成功
    Healthy --> Degraded : 单次健康检查失败
    Degraded --> Healthy : 恢复成功
    Degraded --> Unhealthy : 连续3次失败
    Unhealthy --> Recovering : 等待恢复探测间隔
    Recovering --> Healthy : 恢复探测成功
    Recovering --> Unhealthy : 恢复探测失败
    Unhealthy --> [*] : MCP Server 注销
    Healthy --> [*] : MCP Server 注销
```

### 23.3 编排侧的降级策略

| MCP Server 状态 | 编排侧行为 |
|----------------|-----------|
| Healthy | 正常调用，无感知 |
| Degraded | 编排继续调用，但记录告警日志（warn 级别） |
| Unhealthy | 编排节点标记为"工具不可用"，不发起调用，工作流继续执行但相关节点标记为 Skipped；触发告警通知 |

### 23.4 验收标准

| 编号 | 验收标准 | 验证方法 |
|------|----------|----------|
| AC-MCP-01 | 健康检查周期固定为 30 秒，误差不超过 ±2 秒 | 定时器监控连续 10 次 |
| AC-MCP-02 | MCP Server 连续 3 次健康检查失败后被标记为 Unhealthy | 模拟 3 次失败验证状态转换 |
| AC-MCP-03 | Unhealthy 状态下编排不调用该 MCP Server，相关节点标记为 Skipped | 触发 Unhealthy 后执行编排验证 |
| AC-MCP-04 | MCP Server 恢复后自动重新加入可用列表（探测间隔 60 秒） | 恢复后验证自动重新加入 |
| AC-MCP-05 | 健康检查结果可在编排监控页面查看 | 在监控页面查看 MCP Server 状态 |

| `Node` | 通用工作流节点(LLM/Tool/Condition/Loop/Parallel/Join 等) | `partition_key`、`domain_id`、`id`、`workflow_id`、`type`、`config` | `OrchestrationEntity`、`Workflow`、`Graph` |
| `Execution` | 工作流执行实例 | `partition_key`、`domain_id`、`id`、`workflow_id`、`status` | `OrchestrationEntity`、`Graph` |
| `A2AMessage` | A2A 消息 | `partition_key`、`domain_id`、`id`、`from_agent`、`to_agent` | `Graph` |
| `Agent` | Agent 实体 | 复用 PRD-06(由 PRD-06 定义基础属性) | - |
| `Tool` | Tool 实体 | 复用 PRD-03 | - |
| `MCPServer` | MCP Server 实体 | `partition_key`、`domain_id`、`id`、`name`、`endpoint`、`status` | `Graph` |

> **历史收敛说明**: 原 §27.1 的 `OrchestrationNode` / `AgentNode` / `ToolNode` / `LLMNode` / `ConditionNode` / `LoopNode` / `ParallelNode` / `JoinNode` 八类细分节点已统一收敛为 `Node` + `type` 属性;具体节点类型由 `Node.type` 字段表达,GraphQL / Postgres 表均遵循 §A4 / §26 的统一命名。

### 27.2 关系定义

| 关系类型 | 起点 → 终点 | 含义 |
|----------|-------------|------|
| `WORKFLOW_HAS_NODE` | `Workflow` → `Node` | 工作流包含节点(属性 `order`) |

### 27.3 三标签组合(租户分区与多视图)

为支持多租户分区与跨视图查询(Graph / OrchestrationEntity / Workflow),编排管理模块 Neo4j 节点统一叠加 `partition_key` 维度的视图标签:

```cypher
// Workflow 节点叠加三标签:OrchestrationEntity:Workflow:Graph
// Node 节点叠加三标签:OrchestrationEntity:Workflow:Graph
// Execution 节点叠加双标签:OrchestrationEntity:Graph
```

**视图标签说明**：

| 标签 | 用途 |
|------|------|
| `OrchestrationEntity` | 编排实体视图(供 §A5 GraphQL 跨实体查询) |
| `Workflow` | 工作流节点视图(供拓扑/连接查询) |
| `Graph` | 静态租户分区标签（v6 收束，原 `Graph{partition_key}` 动态标签已统一为静态 `Graph`，Cypher 必须含 `WHERE n.partition_key = $partitionKey` 条件） |

**示例**: `MERGE (w:Workflow:OrchestrationEntity:Workflow:Graph {id: $id, partition_key: $partition_key})`

### 27.4 约束示例

```cypher
CREATE CONSTRAINT workflow_unique IF NOT EXISTS
ON (n:Workflow) ASSERT (n.partition_key, n.id) IS UNIQUE;

CREATE CONSTRAINT node_unique IF NOT EXISTS
ON (n:Node) ASSERT (n.partition_key, n.id) IS UNIQUE;

CREATE CONSTRAINT execution_unique IF NOT EXISTS
ON (n:Execution) ASSERT (n.partition_key, n.id) IS UNIQUE;

CREATE INDEX workflow_domain IF NOT EXISTS
FOR (n:Workflow) ON (n.partition_key, n.domain_id);
```
